# notam-map-data

נתוני מרחב אווירי לאפליקציית Aviation NOTAM Map: אזורים אסורים/מוגבלים מהפמ"ת,
תוספות פמ"ת א-17, סגירות רשות הטבע והגנים — ופיד נוטאמים חי.

הריפו הזה הוא צינור הנתונים: GitHub Actions רץ (יומית לאזורים, כל ~20 דקות
לנוטאמים), מושך מהמקורות הרשמיים, מוודא תקינות, ומפרסם ל־GitHub Pages.
האפליקציה מושכת את `docs/manifest.json` ומעדכנת את עצמה — בלי צורך בגרסת
אפליקציה חדשה.

## מבנה

- `data/` — מקור האמת, ניתן לעריכה ידנית (ארבעה קבצי JSON).
- `docs/` — האתר המפורסם: `manifest.json` + `v1/*.json.gz` + `notams.json`
  (פיד הנוטאמים) + `heartbeat.json` (חותמת הריצה האחרונה).
- `pipeline/` — סקריפטי המשיכה, הוולידציה והפרסום.
- `tests/` — בדיקות pytest.

## פיד הנוטאמים (כל ~20 דקות)

`update-notams.yml` מריץ את `pipeline/fetch_notams.py`: גרידה **אחת** מנומסת
של דף הנוטאמים של רת"א (עם User-Agent מזדהה, לא התחזות לדפדפן) ופרסום
התוצאה כ־`docs/notams.json`. כל האפליקציות המותקנות קוראות את הקובץ הזה
במקום לגרוד את רת"א בעצמן — העומס על אתר רת"א לא תלוי במספר המשתמשים.

- הקובץ מתפרסם כשהנוטאמים השתנו, או כ"פעימת רעננות" כשחותמת `fetchedAt`
  התיישנה מ־3 שעות — כך אפליקציה מבדילה בין פיד שקט לפיד מת (ומעל 4 שעות
  היא חוזרת לגרוד את רת"א ישירות בעצמה).
- רת"א לא זמינה רגעית → הקובץ הקודם נשאר, הריצה ירוקה.
- הדף חזר תקין אבל פוענחו 0 נוטאמים → הריצה נכשלת בקול: מבנה הדף השתנה
  וצריך לתקן את הפרסר.

## התראות לטלפון (ntfy)

שם הערוץ שמור כסוד `NTFY_TOPIC` (הריפו ציבורי — מי שיודע את השם יכול
להציף או לצותת). שלושה סוגי התראות:

- **בוקר, על הצלחה**: הריצה היומית שולחת אישור קצר. יום בלי אישור = הצינור
  תקוע.
- **על כישלון, מיידית**: שם הצעד שנכשל (שמצביע על הסקריפט האשם) ולחיצה
  שפותחת את הלוג. פיד הנוטאמים מתריע רק על הכישלון הראשון ברצף — תקלה
  קבועה לא תצלצל 72 פעם ביום.
- **ערב, תקציר תקלות** (`daily-failure-digest.yml`, ~23:30): כמה ריצות נכשלו
  היום, באיזה תהליך ובאיזה צעד, כולל תקלות שקובץ תהליך שבור לא מסוגל לדווח
  על עצמו. ביום נקי — שקט.

## דופק (heartbeat)

הריצה היומית כותבת `docs/heartbeat.json` עם שעת הריצה, בכל ריצה — גם כשאין
שינוי בנתונים. האפליקציה מציגה ממנו את "בדיקת מקורות אחרונה בשרת" בלי לפנות
ל־GitHub API (שמוגבל לפי כתובת IP, אותה חולקים משתמשי סלולר רבים). תופעת
לוואי מכוונת: קומיט יומי קבוע שמאפס את כיבוי־האוטומט של GitHub לתהליכים
מתוזמנים (60 יום ללא פעילות).

## אוטומציה מלאה

ההרצה היומית (GitHub Actions, ~06:17 שעון ישראל) עושה הכול לבד:

1. מושכת את קובץ הפרק הרשמי של א-17, מפענחת, ו**מחילה שינויים אוטומטית** —
   בלי PR ובלי אישור ידני. הוולידציה עדיין חוסמת פרסום נתונים פסולים.
2. בלם ביטחון: שינוי בנפח חשוד (אזורים רבים נעלמו/השתנו בבת אחת — חתימה של
   שבירת פרסר, לא של עדכון אמיתי) לא מוחל; נשמר קובץ מועמד ונפתח issue.
   ל־PR ידני מלא יש את ה־workflow הישן (`a17-refresh`, מופעל ידנית בלבד).
3. עוקבת אחרי קובץ העדכון המרוכז (ראו למטה) ואחרי קבצי הפרקים.
4. בונה manifest ומפרסמת — האפליקציות מתעדכנות בהפעלה הבאה שלהן.

מה שנשאר ידני: תוכן שאינו גיאומטריה (כללי ב-09, טפסים, נהלים) שחי בקוד
האפליקציה — על אלה נפתח issue בלבד.

## רשת ביטחון: קובץ העדכון המרוכז (קובץ עדכון N-YY)

בנוסף למעקב אחר קבצי הפרקים (`watch_aip.py`), הסקריפט
`pipeline/check_update_file.py` רץ יומית ומאתר את קובץ העדכון המרוכז
שרת"א מפרסמת בעמוד הפמ"ת (העמוד עצמו חסום לבוטים, אבל קבצי ה־PDF לא).
הקובץ מרכז את **כל** עדכוני הפמ"ת, ולכן הסקריפט מסנן: הוא פותח issue רק
כשהעדכון נוגע לתוכן רלוונטי לרחפנים — פרקים א-17 / ב-09 או מילות מפתח
(כטב"ם, רחפן, טיסן, רב להב, BVLOS). עדכון שלא נוגע לרחפנים נרשם בשקט
ב־`data/.aip_update_state.json`. אם הניתוח נכשל, נפתח issue לבדיקה ידנית.

בדיקה ידנית של קובץ מקומי:
`python pipeline/check_update_file.py --pdf <נתיב>`

## עדכון ידני של נתונים

1. לערוך את הקובץ המתאים תחת `data/`.
2. `python pipeline/validate.py`
3. `python pipeline/build_manifest.py`
4. commit + push. כל האפליקציות מתעדכנות בהפעלה הבאה שלהן.

## מקורות ורישוי

- פמ"ת (AIP ישראל) — רשות התעופה האזרחית, gov.il.
- סגירות שמורות טבע — רשות הטבע והגנים.
- הנתונים ציבוריים ומפורסמים כאן עם ייחוס למקור.

**אזהרה: המידע אינו מיועד לניווט מבצעי או לתכנון טיסה. המקור המחייב הוא
הפרסומים הרשמיים של רשות התעופה האזרחית בלבד.**
